# Translation Method

本文件用于长文、章节、整本书、含图材料或译稿审校。短摘录可只使用 `SKILL.md` 的快速规则。

## OODA 工作流

### Observe：识别输入与约束

先看清材料来源、形态和交付目标：

- 来源：PDF 原书、OCR 文本、Markdown、网页摘录、已有中文译稿、图片/表格/流程图。
- 范围：短摘录、单章、连续章节、整本书、译稿审校。
- 目标读者：中文游戏策划、研发、制作人、研究者、发行/商业团队。
- 特殊要求：术语表、图像保留、图内标签中文化、关键英文原句、中英对照、项目内归档。
- 传播边界：个人学习、内部研究、课程材料、公开文章、商业出版或付费内容。

只在这些判断会影响交付方式时继续索取或读取更多材料。否则进入术语层。

### Orient：建立术语与风格

先统一高频概念，再翻正文：

| English | 推荐方向 | 备注 |
| --- | --- | --- |
| verbs | 动词 | 游戏设计语境常指玩家可执行动作。 |
| goals | 目标 | 注意与胜利条件、任务目标区分。 |
| rules | 规则 | 不要随意翻成机制。 |
| systems | 系统 | 可按上下文细化为系统结构、规则系统。 |
| dynamics | 动态 / 动态关系 | 指系统运行后涌现出的关系或变化。 |
| gameplay | 玩法 | 不要机械翻为“游戏性”。 |
| core loop | 核心循环 | 可解释为反复驱动玩家行动的循环。 |
| progression | 成长 / 进程 / 推进 | 依上下文决定，不要全篇混用。 |
| agency | 主体能动性 / 行动自由 | Game Studies 语境下慎译。 |
| uncertainty | 不确定性 | 与 randomness 区分。 |
| randomness | 随机性 | 重点是随机来源或机制。 |
| feedback loop | 反馈循环 | 系统设计语境常用。 |
| difficulty curve | 难度曲线 | 可与节奏、学习曲线区分。 |
| affordance | 可供性 / 行动暗示 | UX 与设计理论语境下谨慎选择。 |
| onboarding | 新手引导 / 上手过程 | 不要机械译成“入职”。 |
| retention | 留存 | 商业与运营语境常用。 |

如果中文社区有多种译法：

- 选更主流、更稳、更适合中文策划阅读的译法。
- 首次出现可写成 `中文（English）`。
- 后文保持一致，不要在同一语境里来回切换。
- 不确定时在术语表中标注“暂定译法”和适用语境。

### Decide：选择翻译策略

按任务类型选择策略：

- 直接翻译：短摘录或单段概念说明。
- 精翻改写：章节、长文、理论段落，重点优化中文信息流。
- 结构化翻译：整本书、连续章节，必须先建文件结构、术语表和来源清单。
- 审校润色：已有译文，优先查术语一致性、英文腔、漏译图注。
- 图像增强：含图材料，图注、图示说明、图内关键标签都进入交付范围。
- 公开传播准备：压缩引用比例、保留来源说明，必要时建议改为摘要、评论或学习笔记。

整本书/PDF 的“完成”必须按质量层级判断：来源拆分、初译、图表回填、中文精校、英文好句摘要、质量门报告。具体标准见 `references/whole-book-quality-workflow.md`。如果只完成初译或合并版，必须报告为“初译完成”，不要称为“完整高质量翻译”。

### Act：翻译与整理

正文处理顺序：

1. 保留标题层级、章节编号、图号、表号。
2. 翻译正文，按中文信息流拆长句。
3. 统一术语，并把新增术语回填到术语表。
4. 翻译图注、表头、流程图标签、脚注和必要的边栏。
5. 对少量关键句用行内英文保留。
6. 补必要的“图示说明”，解释图在论证中的作用。
7. 更新来源清单、待确认术语和审校记录。
8. 对整本书交付，生成 `chapters_polished/` 和 `00-全书精校版.md`，在精校版中保留全部图片引用。
9. 生成 `03-英文好句摘要与译法.md`，用英文短触发词索引精彩表达，用中文摘要和译法点评承载价值。
10. 生成 `audit/质量门报告.md`，列出自动检查、人工复核项和授权边界。

中文风格标准：

- 句子要顺，论证关系要清楚。
- 技术性和可读性同时成立。
- 可以调整语序和拆句，但不要改写作者观点。
- 不用过度口语化，也不要写成机器说明书。

### Evaluate：质量门

交付前至少跑一遍：

- 术语一致性：同一概念是否多译。
- 信息完整性：标题、正文、图注、表格、脚注是否漏掉。
- 图像完整性：图是否裁切得当，标签是否需要中文化。
- 图片引用完整性：图片资产数、分章引用数、合并版引用数是否一致。
- 中文可读性：是否存在明显英文句序、堆括号、过度名词化。
- 英文保留比例：是否只保留真正有价值的原句。
- 英文好句摘要：是否使用短触发词和中文点评，而不是长段原文摘抄。
- 来源与授权：是否记录来源、页码或 URL；公开传播风险是否已说明。
- 项目归档：文件是否落在用户指定项目或文档目录中。

## 来源与版权边界

翻译任务必须先判断交付边界。默认把 copyrighted 长篇材料当作个人学习或内部研究材料，不主动塑造成公开可传播的全文译本。

处理原则：

- 记录来源：书名、作者、版本、章节、页码、URL、截图来源或 OCR 来源。
- 少量引用：英文原句只保留必要的短句，不堆大段原文。
- 公开用途：用户要公开发布时，优先改成摘要、评论、术语解读、读书笔记或局部引用分析。
- 不确定授权：不要声称“可公开商用”；在交付说明中标注需用户自行确认授权。
- 项目内归档：把 `source_manifest.md` 或 README 的来源区作为可追溯入口。

## 图像与表格规范

如果用户说“包括图片”“图也要翻译”“保留图”，不要只放整页截图了事。

处理优先级：

1. 翻译原图注。
2. 补图示说明，说明图在论证中起什么作用。
3. 图内关键标签中文化，尤其是流程图、坐标轴、状态图、系统框图。
4. 裁切到真正 relevant 的图形主体，避免整页截图挤压阅读。
5. 表格保留结构，翻译表头、单位、注释和脚注。

推荐 Markdown：

```md
图 1.2：三扇打开的门，后方各自对应不同内容。

![图 1.2 三扇打开的门（中文化）](assets/fig-1-2.png)

图示说明：这张图展示三种选择背后的结果差异，用来说明结果差异会提升选择的趣味性。
```

如果图内英文只是装饰性文字，不影响理解，可以不强行替换，但要在交付说明中标注。

对整本书，必须建立 `assets/figures_manifest.md`，记录 Figure/Table 编号、PDF 页、原图注、资产路径和回填章节。交付前检查：

- `assets/fig-*.png` 数量。
- 分章 Markdown 图片引用数量。
- 合并版 Markdown 图片引用数量。
- 缺失、重复、坏路径和整页误裁。

如果图内标签尚未中文化，质量门中必须写清楚，而不是默认为完成。

## 英文原句保留规则

默认译成中文。只有符合以下条件之一才保留英文：

- 已是行业名句或用户之后大概率会引用。
- 英文措辞本身有概念辨识度。
- 中文译法会明显损失力度或双关。
- 用户明确要求保留英文或中英对照。

推荐行内嵌入：

```md
Sid Meier 有一句非常著名的话：“游戏是一连串有趣的选择（a game is a series of interesting choices）。”
```

避免：

```md
英文原句：

> "a game is a series of interesting choices"
```

整本书的“好句”需求默认独立成 `03-英文好句摘要与译法.md`：

- 英文只保留 1-3 个词的短触发词，作为索引。
- 中文写“好句摘要”“可借鉴的写法”“可用于哪类设计讨论”。
- 不做逐段双语，不用长块引用。
- 如果用户明确要求原句引用，也只选少量短句，并保留版权边界说明。

## knowledge-vault 与项目目录规则

如果当前工作目录属于 `knowledge-vault` 或用户指定项目：

- 正式交付物放到项目文档区，如 `docs/`、`docs/projects/` 或用户指定目录。
- 不把最终版本散落在 `D:\down`、临时下载目录或未说明的外部目录。
- 新增内容影响可发现性时，同步更新目录页、索引、README 或项目 Gate。
- 如果只是临时抽取、OCR 或裁图，中间文件可以放临时区，但最终文档必须归档。
