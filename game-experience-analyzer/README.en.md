`game-experience-analyzer` is a Codex skill for game design analysis. It does not watch a video and write a loose reaction essay. It first builds screenshot, keyframe, and timestamp evidence, then routes the task into the right lens: early experience, gameplay mechanics, holistic product analysis, MDA, systems-narrative fusion, single-player flow, trailer heat prediction, foresight opportunity, genre strategy, commercialization, or UX/UI.

It is built for game designers, producers, publishing and marketing teams, competitor research, AI-assisted game teams, and anyone who needs to turn "this game feels good/bad" into "where is the evidence, what is the problem, and how should we validate the next step?"

## Why This Exists

Game experience analysis often fails in two ways:

- **Feeling without evidence.** It says "addictive", "bad pacing", or "strong hook" without pointing back to a frame, timestamp, UI element, or loop.
- **One template for every game.** Casual games, single-player games, SLG, X+SLG, trailers, playtest recordings, and screenshot audits all get forced into the same wording.

This skill has a narrow practical goal:

```text
Input media -> Evidence timeline -> Mode router -> Design lenses -> Actionable report
```

It identifies the input source and analysis intent first, then chooses strategy based on genre and sample boundaries. Hook/Loop/Link/Surprise is one available lens, not the universal answer.

## What It Can Analyze

| Input | Best For | Output Style |
| --- | --- | --- |
| Screenshot | UI hierarchy, primary goal clarity, crowded entry points, reward expression, static hook | visual evidence table + UX/experience diagnosis |
| Local recording | first hour, onboarding, tutorials, gameplay loop, monetization interruptions | timeline + event stream + feature ledger |
| Trailer / PV | heat potential, clarity of selling point, conversion path, edit direction | heat potential tier + validation plan |
| Video link | public videos, competitor recordings, cloud clips, social media snippets | access notes + visible evidence + degraded analysis when needed |

## Analysis Modes

| Mode | Use When | Core Questions |
| --- | --- | --- |
| `early_experience` | first login, onboarding, first hour, stickiness | Why does the player continue? Does the first loop close? |
| `gameplay_mechanics` | mechanics, core loop, player actions | What does the player do? How do actions create feedback and growth? |
| `holistic_game_analysis` | whole product, positioning, experience and market together | Do positioning, play structure, content supply, monetization, and window fit each other? |
| `whole_game_mda` | user explicitly asks for MDA | Are Mechanics, Dynamics, and Aesthetics aligned or broken? |
| `systems_narrative_fusion` | theme, value conflict, play-as-narrative | Is the theme generated by mechanics, choices, and consequences? |
| `single_player_design` | single-player flow, level, boss, narrative, open or semi-open structure | Do critical path, pacing, agency, and challenge feedback work? |
| `trailer_heat_prediction` | PV, trailer, ad creative | How strong are the first seconds, selling point, proof of play, platform fit, and conversion bridge? |
| `foresight_opportunity` | opportunity window, project value, whether to follow a direction | What is the opportunity type, remaining window, Go/No-Go condition, minimum validation deadline, and kill condition? |
| `genre_benchmark` | genre strategy, benchmark, X+SLG | Which strategies migrate, and which do not? |
| `problem_diagnosis` | what is wrong, how to fix it | What is the root cause, smallest change, owner, and validation metric? |

## Foresight Window Defaults

Foresight work separates the opportunity validation window from the full development cycle.

| Scope | Default Validation Window | Meaning |
| --- | --- | --- |
| Casual / light games | 1-3 months | Quickly validate direction with assets, prototypes, clicks, retention, or conversion signals. |
| Micro / midcore-heavy games | 3-6 months | Validate core gameplay, retention, monetization bridge, or market conversion before continuing to observe. |
| Large projects | Custom development cycle | Development may take longer, but staged opportunity validation still needs shorter checkpoints. |

## Tooling

| Source | Minimum Setup | Better Setup |
| --- | --- | --- |
| Screenshot | direct visual analysis | OCR when UI text changes judgment |
| Local video | `ffmpeg` for frame extraction | sampled frames + dense event timeline |
| Video URL | browser access or visible metadata | `yt-dlp` equivalent + frame extraction |
| Speech/subtitles | optional | ASR only when voice/subtitle changes judgment |

When tooling is missing, the report keeps `tool_readiness`: missing items, impact, recommended setup, and the degraded scope of the current analysis.

## Evidence Rules

- Every important judgment must point back to a screenshot region, timestamp, keyframe, page metadata, or user-provided fact.
- Screenshots cannot directly prove pacing, feel, waiting time, or loop closure; mark those claims as `uncertain`.
- Trailers/PVs can only predict heat potential and validation paths, not guaranteed sales, revenue, or downloads.
- External research must pass VOI: browse only when it can change genre judgment, version facts, benchmark standards, market heat, window phase, Go/No-Go, or recommendation priority.
- Outputs must distinguish `known`, `unknown`, `uncertain`, `access_notes`, and `tool_readiness`.

## Project Boundary

Temporary collaboration instructions are not public project rules. A user may ask Codex to follow a one-off order, avoid a phrase, use a temporary case, or execute a session-specific sequence; those instructions bind the current task only.

Only write a change into this skill when at least one condition is true:

- It reliably improves analysis quality for other users.
- It is a public, reusable, verifiable method, template, example, or maintenance rule.
- The user explicitly asks to preserve it in the project.

Otherwise, treat it as local collaboration context and do not place it in README, SKILL, references, templates, or examples.
