# 游戏拆解诊断方法

本文件用于用户要求“完整拆解一个游戏”“解释为什么它成立”“判断机制能不能迁移”或“从玩法结构看这个项目”时。它把样本证据转成设计判断，重点不是复述内容，而是找出玩家反复做什么、目标如何驱动动作、系统如何生成动态，以及哪些结构可以被迁移。

不要把本文件当成竞品复刻清单。拆解的输出必须包含迁移边界：哪些是可迁移的结构，哪些依赖题材、美术、社区、平台、时代情绪、内容体量或具体数值，不能直接照搬。

## 1. 样本边界门

先判断当前材料能支撑哪一层拆解：

| 样本范围 | 可判断 | 不可强判 |
| --- | --- | --- |
| 截图 | 静态 UI、可见目标、入口、资源、可能动作、内容承诺 | 操作手感、节奏、循环闭合、长期动态 |
| 3-10 分钟片段 | 局部动作、一次目标推进、短期反馈、局部不确定性 | 完整内容流、长期经济、后期深度 |
| 20-60 分钟录屏 | 首段循环、目标层级、功能引入、早期系统动态 | 完整通关、长期留存、版本后期生态 |
| 多段样本 / full_run_review | 主循环、内容流、阶段变化、系统动态、主题承载 | 仍需区分版本、玩家水平、剪辑和外部资料 |

输出时必须保留 `unsupported_judgment_scope`。如果用户要求“完整拆解”但只有截图或片段，报告可以保留完整框架，但必须把不可判断项标为 `unknown`、`uncertain` 或 `needs_more_sample`。

## 2. 拆解目标

先写 `dissection_goal`，避免无边界地“什么都拆”。

| 目标 | 适合回答 |
| --- | --- |
| understand_success | 这个游戏为什么成立，核心设计支点是什么 |
| find_breakpoints | 哪些地方断裂，为什么不够好玩或不够清楚 |
| transfer_mechanic | 某个机制能否迁移到另一品类、平台或项目 |
| benchmark_genre | 它在品类策略上做对了什么，哪些不能模仿 |
| improve_prototype | 对当前原型给出最小可验证改法 |

`dissection_goal` 会决定报告重点。机制迁移任务必须输出 `transferability_boundary`；问题诊断任务必须输出 issue cards；立项任务必须连接前瞻窗口和验证计划。

## 3. 玩家动词清单

先找玩家反复做的动作，而不是功能名。

| 动词类型 | 观察重点 | 常见误判 |
| --- | --- | --- |
| direct | 移动、瞄准、交换、放置、点击、选择、拖拽等显性动作 | 把动画或按钮数量当成玩法深度 |
| indirect | 玩家动作触发的系统状态变化、资源变化、敌人反应 | 只写操作，不写规则响应 |
| abstract | 预测、规划、权衡、记忆、推理、构筑、估价 | 忽略脑内判断也是核心玩法 |
| interface | 背包、交易、编辑器、卡组、科技树、订单、地图等界面行为 | 把界面当成中性容器，不检查它是否服务循环 |

必须回答：

- 玩家 80% 的操作时间在做什么。
- 最有传播感的卖点是否进入高频动作。
- 动词之间是强化、冲突、顺序、替代，还是互相遮蔽。
- 每个关键动词对应的规则响应是什么。

## 4. 动作和目标对齐

动作只有在目标语境中才有设计意义。拆解时把核心动作连接到目标层级：

| 目标层级 | 观察问题 |
| --- | --- |
| moment_goal | 当前 3-30 秒玩家为什么要这样做 |
| session_goal | 这一局、这一关、这一段为什么要继续 |
| mid_term_goal | 解锁、构筑、基地、路线、章节、活动如何形成进展 |
| long_term_goal | mastery、收藏、排名、表达、社交身份、剧情完成如何让玩家回来 |
| self_authored_goal | 玩家是否容易生成自发挑战、炫技、截图、角色扮演或社区话题 |

质量门：

- 核心动词至少要服务一个短期目标和一个中/长期目标。
- 目标不能只写在文案里，必须能被动作推进。
- 如果目标不改变玩家判断，它只是任务标签，不是有效目标。

## 5. 不确定性校准

不确定性不是“加随机”。先判断玩家应该不知道什么，再看来源是否匹配。

| 来源 | 适合制造 | 风险 |
| --- | --- | --- |
| other_players | 博弈、协作、欺骗、社交声望 | 对弱社交或单机项目成本过高 |
| hidden_information | 探索、侦查、推理、惊喜 | 无反馈时变成迷路 |
| physical_skill | 时机、空间、精度、反应 | 平台和受众可能错配 |
| mental_skill | 预测、规划、资源权衡、构筑 | 过量会导致分析瘫痪 |
| randomness | 变化、重开价值、局势扰动、追赶 | 覆盖玩家努力会损害公平感 |

判断原则：

- 随机性更适合放在局面生成、奖励候选、资源来源或事件扰动里。
- 对玩家已经明确努力换来的结果，随机失败要谨慎。
- 如果玩家无法解释失败原因，不确定性通常是噪音，不是挑战。

## 6. 系统动态图

拆解不止列规则，还要预测规则组合后的玩家行为。

```text
rule_or_input -> player_strategy_shift -> system_response -> experience_dynamic -> risk_or_opportunity
```

至少记录三类动态：

- reinforcing_dynamic：越玩越明确、越熟练、越想继续的正循环。
- tension_dynamic：资源、时间、风险、空间、信息或价值观之间的张力。
- degenerative_dynamic：最优解单一、等待、刷数值、重复劳动、信息过载、失败原因不可读。

动态判断必须引用 evidence_id。只有当当前样本能看到连续行为时，才能高置信判断循环闭合；截图只能写成动态假设。

## 7. 内容流和变化

内容不是越多越好。检查内容是否给核心动词提供新的判断，而不是替代玩法。

| 层级 | 检查问题 |
| --- | --- |
| repeated_content | 重复素材是否改变决策，而不只是换皮 |
| variation_axis | 敌人、地形、订单、牌、角色、目标、资源是否带来新权衡 |
| onboarding_to_mastery | 教学、练习、组合、反转、精通是否有节奏 |
| content_cost | 内容生产成本是否压过系统可复用性 |
| exhausted_point | 玩家何时会看穿系统，什么时候需要新变化 |

如果“内容供给”只是堆关卡、堆角色、堆剧情，而没有改变玩家动词和目标，应标为 `content_substitutes_loop_risk`。

## 8. 受众动机

受众不是年龄、性别或地区标签。拆解要写玩家来这里满足什么欲望：

| 动机 | 可能证据 |
| --- | --- |
| mastery | 高技巧、评分、速度、build 优化、复盘空间 |
| fantasy | 身份、角色、职业、世界观、权力感 |
| expression | 装扮、建造、命名、截图、分享、创作 |
| social_status | 排名、稀有物、关系、协作、社区可见性 |
| relaxation | 低压节奏、熟悉循环、稳定反馈 |
| drama | 冲突、风险、道德代价、叙事后果 |

必须说明受众动机如何被核心动作、目标和反馈承载。不要只写“面向年轻用户”“适合休闲玩家”。

## 9. 可玩主题和意义

主题不能只靠剧情、文案或美术。只有当机制、选择、后果和动态共同承载时，才能说主题是“可玩的”。

检查：

- 玩家被迫权衡的价值是什么。
- 规则奖励了什么、惩罚了什么。
- 玩家做出的选择是否让主题变成后果。
- 循环之外是否能形成复述、讨论、二创或文化对话。

如果当前样本只看到叙事设定，没有看到机制承载，标为 `theme_claim_uncertain`。

## 10. 迁移边界

每次完整拆解都必须输出迁移边界：

| 项目 | 写法 |
| --- | --- |
| transferable_structure | 可迁移的规则关系、动词组合、目标层级或动态 |
| non_transferable_surface | 不能照搬的 IP、角色、剧情、美术、社区梗、数值、运营节奏 |
| dependency | 成立条件：平台、受众、内容产能、社交密度、操作精度、商业模型 |
| adaptation_cost | 迁移到目标项目需要改哪些系统和验证哪些风险 |
| minimum_validation | 最小验证样本、指标、通过线和 kill condition |

## 11. 最低输出

完整游戏拆解诊断包至少输出：

1. `sample_scope_gate`
2. `evidence_index`
3. `dissection_goal`
4. `player_verb_inventory`
5. `action_goal_alignment`
6. `uncertainty_sources`
7. `system_dynamics_map`
8. `content_flow`
9. `audience_desire`
10. `meaning_or_theme`
11. `transferability_boundary`
12. `validation_plan`

如果某节没有足够证据，保留标题并写 `unknown`、`uncertain`、`unsupported_by_sample` 或 `needs_more_sample`，不要补写想象内容。
