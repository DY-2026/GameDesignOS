# Pitch Document Quality Gate

最终文档需要通过可读性、决策性、证据性和执行性检查。

## 5 分钟读者测试

目标读者在 5 分钟内应能回答：

1. 这个游戏一句话是什么。
2. 玩家反复做什么。
3. 为什么这件事有吸引力。
4. 为什么现在值得投入下一步。
5. 下一步要验证什么。
6. 需要多少资源或什么决策。
7. 最大失败风险在哪里。
8. 如果是对外 pitch：当前 proof、预算请求和目标发行/平台 fit 是否足够继续谈。

如果读者必须读完整 GDD 才知道项目为什么成立，文档失败。

## 章节密度门

| 问题 | 修正 |
| --- | --- |
| 世界观很多，玩法很少 | 把设定改写成操作、选择、反馈和后果 |
| 系统很多，核心循环不清 | 删除不服务核心循环的系统 |
| 受众很泛 | 改成行为、动机、拒绝点 |
| 商业判断很确定但没有证据 | 改成 assumption 或 needs_research |
| scope 像正式版 | 切出 MVP、VS、Demo、Release、Post-launch |
| 发行 pitch 只有概念图和设定 | 补 playable build / gameplay video / demo 状态；没有就写 proof-building plan |
| 预算请求和时间线脱节 | 重写 ask、burn/time model 和最小下一步投入 |
| 目标发行方没有匹配理由 | 增加 portfolio/timing/service fit；无法验证则写 needs_recheck |
| 风险只有一句话 | 写风险、影响、缓解、验证和 owner |
| 没有结论 | 补 Decision Request |

## 语言门

- 用评审者能理解的语言，不要堆内部黑话。
- 每个章节开头先给结论，再给展开。
- 表格用于取舍、证据、风险和里程碑，不要把所有内容都做成表。
- 避免万能形容词：丰富、沉浸、有趣、爽快、创新。除非后面能接具体玩家行为。
- 重要判断要能被追溯到 evidence ledger 或 assumption ledger。

## 最终检查

最终文档输出前逐项检查：

- 文档受众和使用场景是否明确。
- 玩家承诺是否在前 20% 篇幅出现。
- 核心循环是否能独立读懂。
- scope gate 是否可执行。
- 商业/平台/发行判断是否标注证据边界。
- 生产计划是否考虑团队能力、内容产能和工具链。
- 对外 pitch 是否写清 proof of play、publisher/platform fit、ask、budget、timeline 和 recheck date。
- 风险是否和验证计划相连。
- Decision Request 是否明确。
