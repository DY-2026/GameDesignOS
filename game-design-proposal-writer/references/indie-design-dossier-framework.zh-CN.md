# Indie Design Dossier Framework

独游设计案的目标不是显得系统完整，而是保护创作锋利度、制作边界和可展示的核心体验。

## 文档主线

```text
创作命题 -> 玩家幻想 -> 商店页承诺 -> 核心动词 -> 最小内容策略 -> vertical slice proof -> 发布/社区验证
```

## 推荐章节

| 章节 | 要回答的问题 | 常见失败 |
| --- | --- | --- |
| Creative Thesis | 这款游戏为什么值得存在 | 只有题材，没有可玩命题 |
| Store-Page Promise | 玩家在商店页看到后期待什么 | 卖点抽象，无法想象玩法 |
| Target Player and Niche | 哪种玩家会被击中，哪种玩家会离开 | 写成泛独游玩家 |
| Player Verbs and Core Loop | 玩家 80% 时间反复做什么 | 只写剧情和氛围 |
| Design Pillars | 哪些原则决定取舍 | 支柱过多，无法砍功能 |
| Content Minimalism Strategy | 少量内容如何产生变化 | 承诺大量关卡、角色、区域 |
| Art and Audio Direction Boundary | 视觉/音频如何服务体验且可制作 | 美术野心拖垮开发 |
| Vertical Slice Plan | Demo 要证明什么 | 做正式版缩小版，不证明独特性 |
| Production Plan | 小团队如何排期和交付 | 不考虑工具链和个人产能 |
| Release and Community Validation | 如何验证传播和购买意愿 | 未证据化地承诺愿望单/爆款 |
| Risk Register | 最可能失败在哪里 | 风险只写时间不够 |

## 独游的范围原则

- 核心体验要尖，不要功能全。
- Demo 证明最独特的 10 分钟，而不是证明正式版所有系统。
- 内容策略优先靠规则变奏、空间复用、组合变化、玩家表达和高质量留白，不靠无限堆量。
- 美术方向应写成可执行边界：资产数量、复用方式、镜头距离、动画复杂度、材质精度、特效数量。
- 音频方向应服务节奏、反馈和情绪，不要写成大量定制曲目承诺。
- 发布策略要标注当前证据。Steam、展会、节日、发行沟通、主播传播、愿望单、众筹等都需要后续调研或真实测试支持。

## 设计支柱建议

独游设计支柱通常 3 条足够。每条要有取舍力。

| 支柱 | 有效写法 | 无效写法 |
| --- | --- | --- |
| 操作支柱 | 每次投递都改变街区规则，让路线规划产生故事 | 好玩、有趣、有沉浸感 |
| 内容支柱 | 复用 12 个街区模块，通过规则卡组合产生新路径 | 地图很多、剧情很多 |
| 情绪支柱 | 温柔但不安全的梦境城市，玩家每次改变规则都承担代价 | 奇幻、治愈、神秘 |

## Vertical Slice 证明点

独游 slice 应证明：

1. 玩家能在商店页承诺和实际操作之间建立期待。
2. 核心动词在 10 分钟内成立。
3. 最小内容集能产生变化。
4. 美术/音频方向可以用目标团队规模做出来。
5. 玩家愿意看下一关、下一局或下一次变化。

不要把正式版系统全部塞进 slice。
