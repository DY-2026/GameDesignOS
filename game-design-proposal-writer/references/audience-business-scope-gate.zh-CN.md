# Audience, Business, and Scope Gate

目标玩家、商业模式和制作范围必须互相支持。策划案要检查三者是否互相打架。

## 目标玩家写法

目标玩家不是人口标签，而是行为、动机和拒绝点。

| 层级 | 要写什么 |
| --- | --- |
| 行为 | 玩家反复做什么、搜索什么、讨论什么、分享什么 |
| 动机 | 他们追求掌控、表达、顿悟、收集、社交、挑战、放松还是叙事沉浸 |
| 情绪收益 | 玩完后希望获得什么感受 |
| 付出方式 | 愿意付出时间、金钱、注意力、学习成本还是创造力 |
| 拒绝点 | 什么会让他们离开 |
| 相邻产品/场景 | 可以从哪些游戏、社区、内容场景观察到他们 |

## 商业模式与体验一致性

| 玩家承诺 | 更自然的商业/发布方式 | 需要警惕 |
| --- | --- | --- |
| 短局爽感、轻策略、碎片时间 | 小游戏、广告、轻 IAP | 广告打断、红点焦虑 |
| 深度构筑、长期成长 | F2P/IAP、赛季、内容更新 | 数值膨胀、付费压迫 |
| 完整叙事和独特审美 | 买断、DLC、原声/设定集 | 内容成本过高 |
| 社交表达和装扮 | IAP 外观、UGC、社区活动 | 审核和内容产能 |
| 硬核挑战和技巧成长 | 买断、赛季挑战、排行榜 | 公平性和外挂风险 |

## Scope 冲突信号

出现以下情况要触发 scope gate：

- 目标平台是短会话，但核心体验需要长时沉浸和复杂学习。
- 商业模式依赖长期 live ops，但团队没有稳定内容产能。
- 卖点依赖高精度美术/动画/物理，但团队没有对应能力。
- 目标玩家追求公平挑战，但商业设计引入强数值付费。
- 独游定位需要锋利表达，但文档堆入大量泛品类系统。
- Demo 需要证明独特性，但开发计划先做底层大系统。

## Gate 输出

每个冲突都要写成：

```text
冲突：
影响：
保留方案：
削减方案：
验证方式：
建议：保留 / 延后 / 替代 / 砍掉
```
