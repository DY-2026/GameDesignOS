# Design Nucleus Options

目的：一句话创意通常不只有一个合理方向。先比较多个设计核候选，再决定要展开哪一个，避免过早把包装层当成完整方案。

## 设计核定义

设计核不是题材、世界观、品类标签或美术风格。设计核必须改变玩家的至少一种行为：

| 行为维度 | 判断问题 |
| --- | --- |
| 选择 | 玩家是否要在有代价的选项之间取舍？ |
| 节奏 | 玩家是否因为时间、压力、资源或机会窗口改变行动节奏？ |
| 成长 | 玩家是否能形成路线、build、关系、地图状态或能力演化？ |
| 表达 | 玩家是否能形成可识别的风格、表演或策略身份？ |
| 反馈 | 玩家是否能清楚感到行为被系统奖励、惩罚或扭曲？ |

## 必填字段

| 字段 | 说明 |
| --- | --- |
| `option_id` | 候选编号，例如 `N1` |
| `nucleus_statement` | 一句话写清玩家反复做什么取舍 |
| `behavior_change` | 改变哪类玩家行为 |
| `promise_fit` | 对玩家承诺的贡献 |
| `platform_fit` | 当前平台假设下是否成立 |
| `production_profile` | 需要的团队能力、内容产能和技术能力 |
| `key_assumptions` | 这个候选依赖哪些 assumptions |
| `smallest_test` | 最小验证方式 |
| `kill_signal` | 什么结果说明该候选应降级或放弃 |

## 比较步骤

1. 从 concept seed 中拆出 2 到 4 个可能的行为冲突。
2. 把每个冲突写成玩家行为句，而不是功能名。
3. 检查每个候选是否能连接玩家承诺、核心循环和验证计划。
4. 排除只改变包装、不改变行为的候选。
5. 选择一个主候选，保留一个备选候选，明确什么时候 pivot。

## 最小示例

```markdown
## Design Nucleus Options

| ID | Nucleus Statement | Behavior Change | Key Assumptions | Smallest Test | Kill Signal |
| --- | --- | --- | --- | --- | --- |
| N1 | 玩家通过照料顺序改变机械植物夜间形态 | 从摆放塔变成规划白天照料优先级 | 玩家能理解照料和变形的因果 | 3 种植物、2 种照料动作、2 波夜袭 | 玩家无法预测变形结果，或只按固定答案操作 |
| N2 | 玩家通过温室布局改写虫群路径 | 从数值堆叠变成空间风险管理 | 单屏网格能提供足够变化 | 6x6 网格、两类虫群、可移动设施 | 每局路线差异不明显 |
```

## 质量门

- 至少提供 2 个候选，除非用户已经明确指定设计核。
- 每个候选必须能写成玩家行为变化。
- 每个候选必须有最小验证方式和 kill signal。
- 不得把“像某游戏”“某题材”“某美术风格”当作候选设计核。
- 主候选的选择理由必须能回连到玩家承诺、scope gate 和 validation plan。
