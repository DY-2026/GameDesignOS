# Prototype Validation Gate

验证计划用于回答：这个创意最小需要做什么，才能知道是否值得继续投入。

不要写“以后测试手感”。必须写清楚测试对象、危险假设、通过标准和失败标准。

## 最小可玩原型

最小可玩原型只验证设计核，不证明完整产品。

它通常包含：

- 一个核心场景。
- 一个核心循环。
- 一种主要失败方式。
- 一种主要奖励方式。
- 少量可选择升级或策略。
- 能记录或观察玩家行为的方式。

它通常不包含：

- 完整剧情。
- 大量角色。
- 完整数值成长。
- 商店和付费。
- 多地图。
- 完整美术。
- 完整新手引导。

## 常用验证指标

| 目标 | 指标 |
| --- | --- |
| 前 10 分钟承诺 | 玩家是否能复述核心冲突，是否愿意再开一局 |
| 选择有效性 | 玩家是否在关键选择前停顿、比较、改变策略 |
| 反馈清晰 | 玩家是否能解释成功和失败原因 |
| 差异化成立 | 玩家是否主动说出“和参照游戏不同的点” |
| 平台适配 | 单局时长、操作疲劳、信息读取、加载容忍 |
| 商业化不破坏体验 | 广告或付费点是否打断核心情绪 |

## 通过标准写法

好标准：

- 5 名测试者中至少 4 名能在 1 分钟内复述核心冲突。
- 首局结束后至少 3 名测试者主动选择再开一局。
- 关键升级选择平均停顿超过 5 秒，且玩家能说出取舍理由。

坏标准：

- 玩家觉得好玩。
- 手感不错。
- 数据表现可以。
- 后续继续优化。

## 失败后的动作

| 失败类型 | 动作 |
| --- | --- |
| 玩家不知道该做什么 | 简化目标和反馈，不先加教程 |
| 玩家不觉得选择重要 | 强化代价、互斥和风险 |
| 玩家只记得题材 | 重提设计核，减少包装 |
| 玩家觉得像竞品换皮 | 增加行为冲突，而非增加世界观 |
| 玩家喜欢但成本太高 | 降 scope，找低成本替代验证 |
