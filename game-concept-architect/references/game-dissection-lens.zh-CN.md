# Game Dissection Lens

本文件用于把“游戏拆解”转成可执行的创意架构和方案审查。它不是读书笔记，也不是竞品复刻方法；它只保留可以进入 skill 工作流的判断门。

核心模型：

```text
玩家动词 -> 规则响应 -> 目标层级 -> 挑战与不确定性 -> 动态 -> 系统关系 -> 内容流 -> 受众动机 -> 可玩主题
```

使用时不要一次性输出所有字段。只有当某一层会改变 `design nucleus`、`scope gate` 或 `validation plan` 时，才展开对应模板。

## 1. 玩家动词

玩家动词不是功能名，而是玩家反复做的事。先找玩家在 moment loop 中真正投入注意力的动作，再判断这些动作是否足以支撑设计核。

| 类型 | 看什么 | 常见误判 |
| --- | --- | --- |
| 直接动作 | 玩家按、拖、点、瞄准、移动、选择什么 | 把 UI 按钮数量当成可玩动作 |
| 间接动作 | 系统因为玩家动作改变了什么状态 | 只写操作，不写规则响应 |
| 抽象动作 | 玩家在脑内观察、预测、权衡、规划什么 | 忽略“判断”本身也是玩法 |
| 界面动作 | 菜单、背包、编辑器、交易、构筑等界面行为 | 把界面当成中性容器，不审查它是否服务循环 |

判断问题：

- 玩家 80% 的操作时间在做什么？
- 这些动词是否会产生不同结果，而不是只是播放不同动画？
- 动词之间是否互相强化、互相冲突或互相遮蔽？
- 最炫的卖点是否真的进入玩家反复动作？

## 2. 动作和目标

动作只有在目标语境中才有设计意义。一个简单动作也可以成立，只要它同时服务多层目标。

目标层级：

| 层级 | 例子 | 审查问题 |
| --- | --- | --- |
| 瞬时目标 | 躲开、命中、连线、交换、抢先完成 | 当前 3 到 30 秒为什么值得思考？ |
| 局内目标 | 过关、达成评分、活到撤离、完成订单 | 这局为什么要继续？ |
| 中期目标 | 解锁路线、构筑成型、完成活动段 | 玩家如何看到进展？ |
| 长期目标 | mastery、收藏、排名、表达、社交身份 | 玩家为什么回来？ |
| 玩家自发目标 | 自定挑战、炫技、截图、角色扮演 | 系统是否容易让玩家生成自己的目标？ |

质量门：

- 核心动词必须至少连接一个瞬时目标和一个局内或长期目标。
- 如果一个目标不能被核心动词推进，要考虑降级、移除或改写核心循环。
- 不要用“丰富目标”掩盖核心动作单薄；目标必须改变玩家判断。

## 3. 不确定性

不确定性用来维持挑战，不是随手加随机。先判断玩家应该“不知道什么”，再决定来源。

| 来源 | 适合制造什么 | 风险 |
| --- | --- | --- |
| 人 | 博弈、社交、交易、欺骗、协作 | 对单机或弱社交项目成本过高 |
| 隐藏信息 | 探索、侦查、推理、惊喜 | 没有反馈时会变成迷路 |
| 身体技能 | 操作精度、时机、空间判断 | 小屏或低操作受众可能错配 |
| 脑力技能 | 规划、预测、资源权衡、构筑 | 过量会导致分析瘫痪 |
| 随机性 | 变化、追赶、局势扰动、重开价值 | 覆盖玩家努力会损害公平感 |

放置原则：

- 随机性更适合放在资源来源、局面生成、奖励候选、事件扰动中。
- 对玩家已经付出明确努力的结果，要谨慎使用随机失败。
- 竞技或高 mastery 项目中，随机性必须可读、可规划或可规避。
- 如果不确定性让玩家无法解释失败原因，它通常不是挑战，而是噪音。

## 4. 动态和阶段

规则是写在设计案里的东西；动态是规则组合后玩家实际经历的东西。设计案必须预测关键动态，而不只列功能。

动态审查：

```text
规则 A + 规则 B -> 玩家策略变化 -> 体验阶段 -> 风险或机会
```

常见动态：

- 位置规则 + 敌人规则 -> 玩家主动管理空间。
- 资源产出 + 消耗压力 -> 玩家形成贪心和保守的取舍。
- 单次限制 + 近距离收益 -> 玩家冒险靠近目标。
- 长期目标 + 短期诱惑 -> 玩家在效率和表达之间摇摆。

阶段审查：

| 阶段 | 触发条件 | 玩家行为变化 | 需要的反馈 |
| --- | --- | --- | --- |
| 初始学习 | 规则首次出现 | 试探、犯错、确认因果 | 清楚的成功/失败反馈 |
| 稳定执行 | 玩家理解规则 | 优化路线、追求效率 | 节奏和收益反馈 |
| 压力变化 | 资源、时间、敌人或机会窗口改变 | 改变策略、接受风险 | 明确的状态变化 |
| 高峰或收束 | 多个系统同时压迫或奖励 | 做关键决策 | 强反馈和后果 |

## 5. 系统关系

系统不是功能文件夹，而是一组互相影响的规则和状态。审查重点是关系，不是数量。

系统关系问题：

- 每个系统改变了哪些玩家动词、目标、不确定性或反馈？
- 系统之间是否有正反馈、负反馈、延迟反馈或追赶反馈？
- 如果只调数值不能解决问题，真正的控制点在哪里？
- 哪些系统关系是 MVP 必须验证的，哪些只是扩展内容？

## 6. 内容流

内容用于重复和变奏核心循环。内容不是越多越好，也不能替代核心循环。

内容流审查：

| 维度 | 问题 |
| --- | --- |
| 变奏来源 | 内容是否来自核心规则的小变化，而不是另起炉灶？ |
| 难度曲线 | 挑战是否随时间变化，并允许多数目标玩家保持投入？ |
| 内容集合 | 单个内容是否服务集合主题、节奏和教学顺序？ |
| 生产方式 | 内容是否能被模块化、复用、组合或工具化生产？ |
| 消耗速度 | 玩家消耗内容的速度是否超过团队产能？ |

## 7. 受众动机

受众不是人口统计标签。要写玩家在这个游戏里想得到什么。

最低审查：

- 玩家想做什么？
- 玩家为什么觉得这件事兴奋？
- 玩家会因为什么反感或离开？
- 参考受众来自哪些相邻游戏或场景？
- 当前创意满足了既有需求，还是试图创造新需求？

如果答案只是“休闲玩家”“二次元玩家”“策略玩家”，还不够。必须写出他们的行为、情绪收益和拒绝点。

## 8. 可玩主题

主题不是文案、设定或价值观宣言。主题必须被操作、选择和后果承载。

审查链：

```text
主题命题 -> 可操作系统 -> 玩家动态 -> 情绪/意义 -> 后果证据 -> 验证指标
```

通过标准：

- 玩家不用读设定，也能通过行动感到主题。
- 主题会影响功能优先级、UI、音乐、美术、关卡或商业化取舍。
- 选择同时有数值代价和意义代价。
- 后果落到人物、空间、秩序、资源或长期状态，而不只是结算文字。

## 触发建议

优先启用本 lens 的情况：

- 用户说“像某游戏”“拆一下这个机制”“这个机制能不能迁移”。
- 创意的玩法动词不清楚。
- 核心循环看似成立，但目标层级、随机性、内容流或受众动机不清。
- 方案包含主题、道德困境、社会系统或“玩法即叙事”。

不必启用全部层级的情况：

- 用户只要非常轻量的一句话判断。
- 输入缺少玩法方向，先做 `concept seed extraction` 更有 VOI。
- 当前最大风险明显是平台、商业、团队能力或 scope，而不是玩法结构。

## 输出门

使用本 lens 后，至少在报告中留下一个可追踪结果：

- `Player Verb Inventory`
- `Action-Goal Alignment`
- `Uncertainty Calibration`
- `System Dynamics Map`
- `Content Flow Plan`
- `Audience Desire Map`
- `Playable Theme Map`

如果没有展开某个结果，说明 VOI 不足或当前证据不支持。
