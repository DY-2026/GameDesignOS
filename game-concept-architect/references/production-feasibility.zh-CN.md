# Production Feasibility

目的：把“想做什么”推进到“能不能做出来、用多久做出来、做出来是否值得”。

核心原则：

- 不能落地的点子不是可行点子。
- 不要把自己不擅长、也没有可靠合作方的能力当作核心卖点。
- 独立游戏要控制时间、成本和内容产能，不要用幻想规模掩盖风险。

## 五个可行性问题

| 问题 | 要判断什么 |
| --- | --- |
| 能力 | 团队是否真的能实现核心卖点？ |
| 技术 | 是否有成熟方案、参考实现、可替代路径或低成本原型？ |
| 内容 | 高光时刻、关卡、剧情、美术、音频是否能持续产出？ |
| 时间 | 项目周期是否可承受？不确定性是否会随时间失控？ |
| 商业 | 价格、销量、成本、回本路径是否至少能形成保守预期？ |

## 能力边界

如果项目的核心卖点依赖某项能力，必须回答：

- 团队内谁能负责？
- 是否已经验证过质量？
- 是否有可靠外包或合作方？
- 如果这项能力失败，项目是否还能成立？
- 是否有更低成本的替代设计？

例外：如果某项能力只是包装或后期增强，可以进入 `Demo 后再做` 或 `宣传概念可以保留但暂不开发`。

## 内容产能检查

许多创意不是死在核心循环，而是死在内容产能。

必须检查：

| 内容类型 | 检查问题 |
| --- | --- |
| 剧情 | 是否能持续写出高质量情境和转折？ |
| 关卡 | 是否能稳定生产并测试足够多的关卡？ |
| 美术 | 是否有风格参考、量级估算和资产复用策略？ |
| 音乐音效 | 是否是情绪承诺的一部分？是否有制作路径？ |
| 高光时刻 | 是否能被设计、复用、组合，而不是偶然出现？ |
| UGC / Mod | 是否有工具链、导入流程、审核和社区预期？ |

## 周期与成本

不要直接写乐观计划。至少写三个层级：

| 层级 | 用途 |
| --- | --- |
| 最小验证成本 | 验证设计核需要多少人、钱、时间 |
| Vertical Slice 成本 | 做出可展示完整体验需要多少投入 |
| 正式版风险区间 | 如果继续做，最大成本和最大周期风险在哪里 |

建议：

- 首款独立项目优先控制周期，不要把计划拉到不可承受长度。
- 预算要按保守收入到中性收入估算，而不是按爆款估算。
- 外包预算、个人机会成本和返工成本都要写入 assumption ledger。

## 商业预期

商业预期不是保证收入，而是用来约束投入。

输出时至少包含：

| 场景 | 含义 | 需要填什么 |
| --- | --- | --- |
| 保守 | 不爆，但有少量认可 | 销量/收入/回本缺口 |
| 中性 | 正常完成并触达目标受众 | 销量/收入/可投入预算 |
| 较好 | 口碑或传播超预期 | 扩展空间 |
| 乐观 | 不作为预算依据 | 只用于 upside，不用于证明可行 |

## 红线

- 不要把“不擅长且没人能兜底”的能力写成核心卖点。
- 不要把大量剧情、关卡、美术、UGC、多人、开放世界同时放进 MVP。
- 不要用爆款收入倒推预算。
- 不要在内容产能未验证前承诺长期高光体验。
- 不要把外部参考当作自己已经具备的能力。
